# Redesign a Product-Team Workflow

Use this entry path to improve a recurring product-team decision or productive motion through responsible human-AI collaboration.

## Entry Modes

- **Guided:** begin with the decision or outcome that must improve.
- **Context dump:** inspect workflow evidence and existing artifacts before asking gaps.
- **Best guess:** label assumptions, reduce automation and scope, and establish a baseline before redesign.

## Sequence

1. Run [`aipom-workflow-opportunity-advisor`](../skills/aipom-workflow-opportunity-advisor/SKILL.md) when the priority workflow is unclear.
2. Run [`aipom-productive-motion-map`](../skills/aipom-productive-motion-map/SKILL.md) against observed cases.
3. Use [`human-aipom-work-contract`](../skills/human-aipom-work-contract/SKILL.md) to assign human judgment, AI responsibilities, review, authority, and fallback.
4. Add context, behavior, evaluation, autonomy, and accountability components where the consequences require them.
5. Run [`aipom-decision-cycle-redesign`](../skills/aipom-decision-cycle-redesign/SKILL.md) for an integrated pilot.
6. Use [`aipom-workflow-playbook-builder`](../skills/aipom-workflow-playbook-builder/SKILL.md) only after representative use produces credible evidence.

## Context Handoff

Preserve observed cases, baseline measures, decisions, actors, inputs, handoffs, delays, rework, exceptions, failures, context, responsibilities, evaluations, controls, outcomes, burden, owners, and open questions.

## Stop and Escalation Rules

- Do not automate the documented ideal instead of observed work.
- Do not optimize output speed while hiding review burden or decision quality.
- Keep consequential decisions human when authority or evidence is unclear.
- Do not standardize an untested redesign.

## Completion

Finish with an observed baseline, redesigned decision cycle, explicit responsibilities and authority, representative pilot evidence, measures and countermeasures, an adoption decision, a named owner, and a reusable playbook when earned.
